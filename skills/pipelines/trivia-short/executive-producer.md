# Executive Producer - Trivia Short Pipeline

## When To Use

The user references the daily-trivia workflow. Triggers:

- "process <slug>" / "new video is ready" / "the clips are in the library"
- "render row <N>" / "publish row <N>"
- "process feedback for <slug>"
- A bare slug like `australia-wider-than-moon` after recent context establishes
  the trivia workflow

If the user is asking for a one-off short or a non-trivia video, pick a
different pipeline. This pipeline is specific to the Post Calendar workflow.

## Philosophy

Trivia-short is **sheet-authoritative**: the Posts sheet is the source of
truth for every input to the pipeline. Artifacts under
`projects/<slug>/artifacts/` are *derived views* of the sheet row — they can
be regenerated by re-reading. Only renders (`projects/<slug>/renders/`) are
data the pipeline produces that doesn't come from the sheet. Your job:

1. **Treat the sheet as authoritative on every entry.** Re-read the row at
   the START of each run. If the row hash has changed since the cached
   artifact was written, the artifact is stale — re-run from idea.
2. **Run the user's house defaults.** Per memory: assembly is always full
   production (`--with-vo --with-music --with-sfx --silent-hook`); publish
   requires local approval; every render is frame-reviewed before "ready".
3. **The feedback loop is not a stage.** It's a re-entry decision. When asked
   to process feedback, classify the note in `feedback.json` and re-enter the
   right stage.

## Sheet Contract

The sheet owns the data. The pipeline owns the renders + transient cache.
This table is the source of truth for who reads/writes what.

> **Layout note.** Post-2026-05 cleanup removed 10 columns, then the
> later Style cleanup deleted column N (Visual Style) since every row
> renders photorealistic. Surviving columns shifted left each time.
> Posts is now A:Z (26 columns). Code should resolve columns via
> `post_row.cell_for(sheets, row, field)` — the table below lists the
> *current* letter for orientation only.

| Col | Field | Read by | Written by |
|---|---|---|---|
| C | Mode (Facts / Choices) | `idea` | — (human) |
| D-K | Script copy (Topic, Hook, Question, Answer Prompt, Ending, Resolution, CTA, Trivia UID) | `idea`, `script` | — (human) |
| L | Final Status | `executive-producer` (loop entry) | `publish` (writes "Ready to publish") |
| M | Final Video Link | `publish` (re-publish: detect existing) | `publish` |
| N | Reaction Archetype | `idea`, `asset` | — (human) |
| O | Reaction Prompt (VLOOKUP) | `script` (for VO/prompt context) | `idea` (picker patches the hardcoded Clip ID) |
| P | Reaction Filename (VLOOKUP) | `asset`, `edit` (assemble script) | `idea` (picker patches the hardcoded Clip ID) |
| Q | Hook Emphasis Override | `edit` | `edit` (writes resolved word back) |
| R | Body Prompt | `script` | — (formula) |
| S | Body Filename | `asset`, `edit` | — (human) |
| T | Closer Prompt | `script` | — (formula) |
| U | Closer Filename | `asset`, `edit` | — (human) |
| V | TikTok Description | `publish` | — (human) |
| W | Pinned Comment | `publish` | — (human) |
| X | Hero Visuals | — | — (human) |
| Y | Slug | `idea` (resolves project path) | — (human) |
| Z | Music Track | `edit` | — (human) |

**Stages only ever write four columns: L/M (`publish`), Q/Emphasis
(`edit` via `assemble_modular.py`), and the Reaction Prompt/Filename
formulas (`idea` via `pick_reactions_llm.py`, which patches the hardcoded
Clip ID inside the existing VLOOKUP formula).**
Every other column is human-owned. Never write prompts or other columns from
the pipeline.

Earlier sheet versions had Final Feedback as an inline column; that
column was removed in the 2026-05 cleanup. Per-row review feedback now
lives in `projects/<slug>/artifacts/feedback.json`, written by the web
app's `/api/feedback` endpoint.

When the reviewer clicks Save & Re-render, the render pipeline runs
**Phase 0: feedback_router.py** which calls Claude with the feedback text +
current `words.json` + sheet row, and emits a structured
`feedback_plan.json` with classifications + patches. The plan drives:

| Classification | Patch type | Effect |
|---|---|---|
| `vo_length` | `allow_shorten_vo` | `shorten_vo.py` retry path uses feedback text as Claude prompt context |
| `caption_word` | `set_word(target_word, near_time_ms, new_word)` | `apply_feedback_patches.py --phase post` edits `words.json` after reconcile, before Remotion render. Matched by word + nearest timestamp so the patch survives transcribe re-runs that shift indices |
| `caption_timing` | `set_timing(target_word, near_time_ms, new_start_ms, new_end_ms)` | Same; nudges `startMs`/`endMs` |
| `brand_token` | `add_brand(token)` | Pre-phase: writes `brand_tokens_extra.json`; `transcribe.py` unions it with the static `BRAND_TOKENS` |
| `music_volume` | `set_music_volume_db(value_db)` | Pre-phase: writes `assemble_overrides.json`; the assemble flags pick it up |
| `clip_change` | `regenerate_segment(segment)` | Pre-phase: writes `feedback_blockers.json` and surfaces in the job log — human must regenerate in OpenArt |
| `unclear` | (none) | Surfaced in `feedback_plan.unresolved[]` for the human |

The Clips library spreadsheet (`CLIPS_SHEET`) is read-only from this
pipeline. Adding new reaction clips, marking them `Generated`, or editing
personas is a human job done in the catalog itself.

## Sheet Sync At Stage Entry

Before any director starts work:

1. Re-read the row (all columns the stage needs per the table above).
2. Hash the relevant cells. Compare against `metadata.sheet_revision` in the
   most recent cached artifact for this slug.
3. If hash differs → the human edited the row between runs. Invalidate ALL
   downstream artifacts and re-enter at `idea`.
4. If hash matches → reuse cached artifacts as-is, run only the requested
   stage.

Hashing rule: SHA-1 of the JSON-encoded list of the cells the pipeline reads
(see the table). Store the hash in each artifact's `metadata.sheet_revision`.
Idea stage is the canonical hash-writer.

## Feedback Re-entry Decision Tree

When the user says "process feedback" or names a slug with a feedback note:

1. Read Final Status for the row (column L) and `projects/<slug>/artifacts/feedback.json`
   (written by the web app's review panel).
2. If L = `Approved` or feedback.json is absent/empty, nothing to do.
3. Classify the feedback note:

| Pattern in feedback | Re-enter at | Why |
|---|---|---|
| Approval phrase ("good to go", "ship it", "approve") | n/a — flip L=`Approved`, clear feedback.json | Done |
| Caption text fix ("change `<word>` at <ts> to `<word>`", "fix typo") | `compose` | Patch words.json, re-render Remotion |
| Caption timing/styling ("highlight wrong word", "captions overlap") | `compose` | Adjust TriviaWithBg constants or words.json |
| VO copy change ("change the CTA to ...") | `script` -> `edit` -> `compose` | VO is regenerated in assemble_modular |
| Music/SFX change ("music too loud", "remove tick at 5s") | `edit` -> `compose` | Re-run assemble_modular with adjusted flags |
| Reaction-fit change ("wrong persona", "reaction doesn't match the closer") | `idea` -> all downstream | Re-run `pick_reactions_llm.py --row <N> --apply`, then re-render |
| Body/closer clip change ("body clip doesn't match", "regenerate closer") | `assets` -> `edit` -> `compose` | Regenerate the named segment via OpenArt |
| Reaction archetype change ("change archetype") | `idea` -> all downstream | Archetype is brief-level; picker re-runs and the Reaction Prompt/Filename formulas get re-patched |

If the feedback is ambiguous, ask the user to clarify which axis (caption /
VO / music / clip / style) before re-entering.

After processing: re-run the affected stages and write the new render through
`publish`. When you re-render via the web app's Save & Re-render button, the
pipeline auto-clears `feedback.json` + `feedback_plan.json` at the end of the
run (race-safe: only if the reviewer hasn't added newer feedback during the
render). If you're processing feedback from the conversation rather than the
web UI, clear those two files yourself when the fix is in. Persistent
artifacts — `brand_tokens_extra.json`, `assemble_overrides.json`,
`feedback_blockers.json` — stay; they're project state, not the conversation.

## Budget & Time

- `budget_default_usd: 0.30` — trivia is cheap (transcription + small VO +
  music gen + SFX library). Anything over this is unusual — flag and ask.
- `max_wall_time_minutes: 20` — Remotion render is the slowest step (~3-5min).
  If a single stage exceeds 10min wall clock, abort the stage and ask the user.

## Decision Communication Contract

Per `AGENT_GUIDE.md`, announce before paid calls. For trivia-short:

- **Asset stage (automated_openart mode):** announce model + variant count
  before driving OpenArt — each generation costs OpenArt credits.
- **Edit stage:** announce that you're applying the four default flags. If
  the user has asked you to deviate (e.g. `--no-music` for a noisy day), say so
  explicitly in the announcement.
- **Compose stage:** announce the Remotion composition and target output path.
- **Publish stage:** announce the row and Drive folder, then **wait for human
  OK**. Never auto-run publish.py — this is a hard rule from user memory.

## What Not To Do

- Do not auto-publish. Publish always waits for explicit human approval.
- Do not skip frame-review at compose. The render is not "ready" until you
  have visually verified captions at key beats.
- Do not invent feedback resolutions. If `feedback.json` is unclear, ask.
- Do not write to the sheet outside the publish stage (except U emphasis
  override, which `assemble_modular.py` owns).
- Do not bypass the assemble defaults without telling the user.
